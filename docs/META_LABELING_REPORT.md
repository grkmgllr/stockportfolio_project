## 1. Executive Summary

This report documents the design and implementation of a **Meta-Labeling** pipeline that extends our existing PyTorch-based time-series forecasting system into a complete algorithmic trading framework. The architecture follows Marcos Lopez de Prado's methodology from *Advances in Financial Machine Learning* (2018), where a **primary model** (deep learning) predicts the direction of the market, and a **secondary model** (gradient boosting) learns to *filter* those signals by predicting the probability that each trade will be profitable.

The key contribution is a clean separation of concerns: the deep learning models (TimeMixer, TimesNet) handle the difficult problem of price-level forecasting, while a LightGBM classifier handles the decision of *whether to act* on each prediction. This two-stage approach is shown in the literature to improve Precision and risk-adjusted returns without sacrificing the primary model's architecture.

The implementation adds **5,625 lines of Python** across **57 files**, introducing 6 new modules while maintaining full backward compatibility with the existing codebase.

## 2. Motivation and Theoretical Background

### 2.1 The Problem with Raw Predictions

Standard regression-based forecasting models output continuous price predictions (e.g., "tomorrow's High will be $215.30"). Converting these into binary trading decisions ("buy" or "don't buy") using a fixed threshold is suboptimal because:

- The model's accuracy varies across market regimes (high volatility vs. low volatility).
- Not all correct-direction predictions are equally profitable.
- A single threshold cannot capture the confidence distribution of the model.

### 2.2 Meta-Labeling (Lopez de Prado, 2018)

Meta-Labeling solves this by decoupling the problem into two stages:

1. **Primary Model (Side):** Decides the direction — "I predict the price will go up." This is our TimeMixer/TimesNet deep learning model.
2. **Secondary Model (Size):** Decides the confidence — "How likely is this prediction to result in a profitable trade?" This is our LightGBM classifier.

The secondary model is trained on **binary labels** generated by the Triple Barrier Method, which objectively evaluates whether each historical prediction would have been profitable. The key insight is that the secondary model does not predict prices — it predicts the *quality* of the primary model's signal, using market context features (volatility, momentum) that the primary model may not explicitly capture.

### 2.3 The Triple Barrier Method

Each trade is evaluated against three exit conditions:

| Barrier | Description | Label |
|---------|-------------|-------|
| **Upper (Take-Profit)** | Price reaches the predicted High | 1 (success) |
| **Lower (Stop-Loss)** | Price falls below a volatility-based threshold | 0 (failure) |
| **Vertical (Timeout)** | Maximum holding period expires | 0 (failure) |

The label records which barrier was hit *first*, creating a ground-truth dataset for the secondary model.

### 2.4 Why Purged Cross-Validation?

Financial time series violate the i.i.d. assumption of standard K-Fold cross-validation in two ways:

1. **Overlapping event windows:** A trade that starts in the training set and ends in the test set leaks future information.
2. **Serial correlation:** Market reactions to events in the test period can persist into adjacent training observations.

Purged K-Fold (Lopez de Prado, Ch. 7) addresses both by removing (purging) overlapping training observations and adding a time embargo after each test fold.

## 4. Implementation Details

### 4.1 Data Pipeline: Minute-Bar Parquet Integration

**Problem:** The original pipeline used daily OHLCV data from Yahoo Finance (5 features). We acquired 10 years of minute-level bar data (1,804,001 rows) for AAPL from a professional data provider, stored as Parquet.

**Solution:** We built `scripts/resample_parquet.py` that:

1. Reads the Parquet file using `pandas.read_parquet()`.
2. Filters to post-January 2021 data (discarding pre-COVID periods that may bias the model).
3. Resamples to daily bars using standard market aggregation:

| Column | Aggregation Rule |
|--------|-----------------|
| Open | First bar's open of the day |
| High | Max of all intraday highs |
| Low | Min of all intraday lows |
| Close | Last bar's close of the day |
| Volume | Sum of all intraday volumes |
| VWAP | Volume-weighted mean: `Σ(vwap × volume) / Σ(volume)` |
| Transactions | Sum of all intraday transaction counts |

4. Saves as CSV with capitalized column names matching the existing pipeline convention.

**Result:** 1,276 daily bars with 7 input features (up from 5).

**Backward Compatibility:** The `YahooDataset` class auto-detects whether the CSV has 5 or 7 columns. The model configuration reads `enc_in` from the dataset at runtime, so no hardcoded values need updating.

### 4.2 Triple Barrier Method (`trading_logic/triple_barrier.py`)

The implementation provides two functions:

**`get_daily_volatility(close, lookback=20)`**
- Computes rolling standard deviation of log returns.
- Used to dynamically size the stop-loss barrier.

**`apply_triple_barrier(df, pred_high_col, ...)`**
- **Upper barrier:** Set directly to the primary model's predicted High price. This evaluates the specific price-level forecast rather than using a fixed percentage.
- **Lower barrier:** Dynamic, computed as `entry × (1 − max(daily_vol × sl_multiplier, min_sl_pct))`. Widens in volatile markets, tightens in calm ones.
- **Vertical barrier:** Maximum holding period (configurable, default 5 bars).
- **Event tracking:** Records `t_start` and `t_end` for each trade, which PurgedKFold uses to detect overlapping observations.

### 4.3 Feature Engineering (`scripts/generate_meta_labels.py`)

This script bridges the primary and secondary models. It produces 11 features:

| Feature | Category | Description |
|---------|----------|-------------|
| `atr` | Volatility | Average True Range (14-bar) |
| `rolling_vol` | Volatility | Rolling std of simple returns (20-bar) |
| `daily_vol` | Volatility | Rolling std of log returns (20-bar) |
| `rsi` | Momentum | Relative Strength Index (14-bar) |
| `macd_line` | Momentum | EMA(12) − EMA(26) |
| `macd_signal` | Momentum | 9-bar EMA of MACD line |
| `macd_hist` | Momentum | MACD line − signal (divergence) |
| `pred_return` | Model signal | `(pred_high / close) − 1` |
| `pred_close_return` | Model signal | `(pred_close / close) − 1` |
| `Vwap` | Microstructure | Volume-weighted average price |
| `Transactions` | Microstructure | Number of trades per bar |

The prediction-derived features (`pred_return`, `pred_close_return`) are particularly important: they capture *how ambitious* the primary model's forecast is, giving the secondary model a sense of the signal's magnitude.

### 4.4 Purged K-Fold Cross-Validation (`trading_logic/purged_cv.py`)

**`PurgedKFold` class** — scikit-learn compatible cross-validator.

**Purging algorithm:**
For each test fold spanning `[test_start, test_end]`, a training observation at index `i` is removed if its event window overlaps with the test period:

```
purge if: t_start[i] ≤ test_end  AND  t_end[i] ≥ test_start
```

**Embargo:** After purging, observations in the range `[test_end + 1, test_end + n_embargo]` are also removed to guard against serial correlation.

**Verification:** We ran an assertion-based smoke test with 100 observations, 5-bar event windows, and 3-bar embargo. All overlap and embargo leak checks passed, with training sets reduced by 3–8 observations per fold (as expected).

### 4.5 LightGBM Meta-Classifier (`models/meta_classifier/lightgbm_model.py`)

**`MetaClassifier` class** wraps `lightgbm.LGBMClassifier` with:

- **Built-in PurgedKFold:** The `fit()` method internally creates a `PurgedKFold` splitter and trains one model per fold with early stopping.
- **Probability output:** `predict_proba()` returns confidence scores in [0, 1], not hard classifications. This enables downstream position sizing.
- **Class imbalance handling:** `is_unbalance=True` by default, which tells LightGBM to weight the minority class more heavily.
- **Final model retraining:** After CV, the model is retrained on the full dataset using the average best iteration count from the folds.

**Default hyperparameters:**

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| `n_estimators` | 500 | High ceiling, controlled by early stopping |
| `learning_rate` | 0.05 | Conservative to prevent overfitting on small datasets |
| `num_leaves` | 31 | Standard for tabular data |
| `subsample` | 0.8 | Row subsampling for regularisation |
| `colsample_bytree` | 0.8 | Feature subsampling for regularisation |
| `reg_alpha` / `reg_lambda` | 0.1 | L1/L2 regularisation |

### 4.6 Evaluation Metrics (`trading_logic/evaluation.py`)

**Classification metrics:**

- **Baseline:** All signals taken → Precision = base rate, Recall = 1.0.
- **Filtered:** Only high-confidence signals taken → Precision should increase.

**Probabilistic Sharpe Ratio (PSR):**

Implements Lopez de Prado (2014). The PSR answers: "What is the probability that the observed Sharpe Ratio exceeds a benchmark, after correcting for skewness and kurtosis?"

$$PSR = \Phi\left(\frac{(\hat{SR} - SR^{*})\sqrt{T-1}}{\sqrt{1 - \hat{\gamma}_3 \hat{SR} + \frac{\hat{\gamma}_4 - 1}{4}\hat{SR}^2}}\right)$$

Where $\hat{\gamma}_3$ is skewness, $\hat{\gamma}_4$ is excess kurtosis, and $\Phi$ is the standard normal CDF. A PSR > 0.95 indicates the strategy's performance is unlikely due to chance.

## 7. Known Limitations and Future Work

### Current Limitations

1. **Primary model accuracy:** The TimesNetPure model's MAE of ~$48 on a ~$200 stock means predictions are significantly offset from actual prices. This causes nearly all trades to be labeled positive (95.7% base rate), limiting the meta-classifier's discriminative power.

2. **Small dataset:** With only 1,276 daily bars and a 70/15/15 split, the test set contains only 158 samples (138 after warm-up). More data or alternative timeframes would improve statistical reliability.

3. **Single asset:** The pipeline currently processes one ticker at a time. Multi-asset portfolio logic is not yet implemented.

### Planned Improvements

1. **Model tuning:** Hyperparameter optimisation of the primary models (TimeMixer, TimesNetPure) to reduce MAE and produce more realistic price-level forecasts.

2. **Multi-timeframe features:** Leveraging the minute-bar parquet data at multiple resolutions (hourly, 15-min) to create multi-scale features for the meta-classifier.

3. **Position sizing:** Using the meta-classifier's probability output to scale position sizes (Kelly criterion or fractional Kelly).

4. **Walk-forward validation:** Replacing the fixed train/val/test split with an expanding-window walk-forward scheme for more realistic backtesting.

5. **Multi-asset portfolio:** Extending the pipeline to handle multiple tickers simultaneously with portfolio-level risk management.
